# 团队分工：下一阶段优化（六人独立分工）

本文件给出针对 `AGENT_STAGE_SUMMARY.md` 中“下一阶段优化目标”的详细六人独立分工说明。每个任务互相独立，包含目标、工作内容、关键交付物、验收标准、建议修改文件、预估工时与风险缓解方案。

---

## 按前后端分工概要

为便于执行与验收，以下将原有六人分工按“后端任务 / 前端任务”进行汇总：

- 后端任务（主要负责服务、模板、规则、数据与稳定性）：成员 A、B、C、D、F
	- 成员 A：Prompt 模板库、mock tryon 服务、按部位（`target_garment`）的 inpainting/input 构建与版本化。
	- 成员 B：推荐规则引擎（`rules_engine.py`）、候选生成与可解释性输出格式、mock 推荐端点。
	- 成员 C：反馈数据模型（`UserFeedback`）、反馈接收 API、离线重排脚本与迁移脚本支持。
	- 成员 D：重试中间件、错误分级、结构化日志（含 `prompt_version`/`target_garment`/`mask_used` 等字段）。
	- 成员 F：迁移脚本、接口契约文档维护、交付与部署说明（不承担模块功能测试）。

- 前端任务（主要负责交互、展示、埋点与降级体验）：成员 E（并与后端各成员对接契约）
	- 成员 E：试穿界面部位选择控件、`target_garment` 字段传递、mock 支持、降级与重试 UI、多候选展示与反馈交互。

说明：每位后端成员须在其交付中包含对应的接口契约与示例请求/响应；每位成员同时负责自身模块的单元测试与一项差异检测集成测试，保证“非目标部位未被修改”的要求。

目标：提升试穿图像质量与一致性，产出可复用的 Prompt 模板库与本地可执行的模拟器。

- 设计并实现按衣物类别（上衣/下装/裙/配饰）和目标替换部位（`target_garment`: `top`/`bottom`/`dress`/`full`）的参数化 Prompt 模板（JSON/YAML），移除对人物姿态的修改要求（姿态保持不变）。
- 构建一个轻量化“本地模拟器”模块，用于在没有真实 AI 网关时用规则/占位图替代以便测试（位置建议：`backend/services/mock_tryon.py`）。
- 为 Prompt 提供版本管理（模板带 `version` 字段）、示例输入/输出与性能说明文档。
- 为 Prompt 提供版本管理（模板带 `version` 字段）、示例输入/输出与性能说明文档，并在模板中加入 `strictness`、`mask_hint`、`target_garment` 等字段以支持按部位精确编辑与重试策略。

关键交付物：
- `prompt_templates/{category}.yaml`（或 JSON）一套模板，包含示例参数与说明。
- `backend/services/mock_tryon.py`（本地模拟器）与调用示例脚本 `scripts/run_mock_tryon.py`。
- Prompt 使用文档与质量评估报告（包含 5 个测试样例）。

- Prompt 模板覆盖至少 4 类衣物与 3 种姿态；
- 人工评审通过率 ≥ 80%（示例集）。
 验收标准：
 - 模拟器在本地可运行并生成可视化结果；
 - Prompt 模板覆盖至少 4 类衣物与 `target_garment` 四种选项（`top`/`bottom`/`dress`/`full`）；

 - 人工评审通过率 ≥ 80%（示例集）。

预估工时：4-6 个工作日。

风险与缓解：若真实 AI 返回行为不可控，使用模拟器与清晰版本化约定减少阻塞。

## 成员 B — 推荐规则与候选生成负责人（多候选 + 可解释性）

目标：实现独立的推荐规则引擎，返回 N 个候选与简短可解释理由，能通过 HTTP mock 接口进行验证。

主要工作：
- 在 `backend/app/services/` 新建 `rules_engine.py`，实现基于颜色/风格/场景的规则集合与权重化排序。
- 设计一个候选输出格式（示例 JSON），包含 `candidate_id`, `score`, `reason` 字段，并提供 mock HTTP endpoint（例如 `/internal/mock_recommend`）。
- 编写规则配置文件（YAML）并支持热加载（供本地调试）。

关键交付物：
- `backend/app/services/rules_engine.py`、`config/rules/*.yaml`、`scripts/run_mock_recommend.sh`
- 输出示例与单元测试（覆盖规则优先级与排序）。

验收标准：
- Endpoint 返回 N（可配置，默认 3）个候选，且每个候选包含 `reason` 字段；
- 规则单元测试覆盖率 ≥ 85%。

预估工时：5 个工作日。

风险与缓解：若需要实时用户画像，先实现静态规则并暴露扩展点供成员 C（反馈）后续接入。

## 成员 C — 用户反馈采集与离线重排负责人（反馈埋点 + 数据模型）

目标：独立实现反馈数据模型及采集 API，能够离线生成重排结果以供评估（不直接依赖实时前端）。

主要工作：
- 在 `backend/app/models.py` 增加 `UserFeedback` 模型（或独立表），并提供 Alembic 迁移脚本（可在本地 SQLite 上验证）。
- 实现反馈接收 API `/internal/feedback`（内部使用 mock 前端数据），记录事件类型（`like`, `save`, `click`, `view`）与时间戳。
- 编写离线重排脚本 `scripts/offline_reorder.py`，读取反馈数据并输出重排候选列表供验证。

关键交付物：
- 数据模型与迁移脚本、反馈收集 API、离线重排脚本与示例数据集（CSV/JSON）。

验收标准：
- 在本地运行迁移并能插入/查询反馈记录；
- 离线重排脚本能基于反馈数据改变候选排序（提供对比示例）。

预估工时：4-6 个工作日。

风险与缓解：若生产 DB 不允许迁移测试，使用本地 SQLite 与迁移脚本确保可复用。

## 平均分配与独立推进（更新说明）
